VOSpec Tool For Handling VO Spectra through SSAP

 

 

 

HOW TO PUBLISH YOUR DATA THROUGH SSAP

 

The procedure involves, basically, two steps:

 

  1. Create an SSAP server “wrapping” your data

  2. Have your spectra in a format understandable by the SSAP protocol

 

 

1. SSAP Wrapper around your (remote access) data

 

Although there is no fixed specification for the SSAP yet, we give here the one we have been working with, which is a very simple and high level specification without too much detail. Only a very limited set of parameters is requested, among which is our dimensional analysis parameters from Osuna-Salgado (see presentation at the ADASS XIV at: http://adass.ipac.caltech.edu/Presentations/Tuesday/Osuna_O5_3.pdf for a brief description of these parameters. A more detailed description is on the way).

 

In order to have your data available through SSAP, you have to build a URL service that accepts the parameters POS and SIZE as input. The format for the values is:

 

POS=RA,DEC, e.g.: POS=304,38

SIZE=size, e.g.: SIZE=1

 

The RA and DEC POSition are given in decimal degrees and the size is given in degrees (this is similar to the IVOA SIAP specification).

 

The output of this query has to be a VOTable with pointers (CDATA) to your spectra (e.g., a URL like: http://<yourURL>/spectrum_010101.fits). This is also very much like in the SIAP protocol.

 

The following is the part of the VOTable output file where the output FIELDs are specified. I have written a word MUST, SHOULD or OPTIONAL depending on whether the parameters are compulsory or not for the VOSpec to be able to work properly with the spectra. Those certainly compulsory are the dimensional parameters. The future SSAP specification will format them properly, but for the time being they should appear as follows: (the example following comes from our ISO SSAP service, hence the optional keywords like ObsId, etc.)

 

<FIELD ID="ObsId" ucd="OBS_ID" datatype="char" arraysize="*"/>

SSAP wrapper around your local data


 


If you happen to have some local -or remote- data but
do not have an SSAP service running and do not dear to build one, you
can still wrap your data in an SSAP-VOTable like form, so that they
can be seen in the VOSpec together with remotely available real SSAP
service spectra. We provide, within the VOSpec tool, a “Wrapper
Creator” that helps you in doing this. 



The process would be as follows: let’s say you
have some data available at your local directory at location:


 /home/posuna/IVOA/HD161796/spectrum


(the location of the files can equally be a URL)


 


And the spectra you want to show are called:


 


2mass_votable.xml 



BVRI_votable.xml 



IRAS_votable.xml 



psp10601706.fits


 


(note that three of them are VOTable spectra
(including a photometry one) and one is a fits file).








 


 


then, you would build your wrapper SSAP file by
invoking the WrapperCreator at the lower right 



corner of the VOSpec. 



    
	
  1. Click on Wrapper Creator.
    
	
  2. Introduce the wrapper filename (will be your
	local SSAP wrapper filename).
    
	
  3. Select one of the local (or remote) spectrum you
	want to be able to display in VOSpec. For example, IRAS_votable.xml
    
	
  4. Choose the file type. In this case, the file
	corresponds to a VOTABLE spectrum, therefore choose VOTABLE.
    
	
  5. Input the Wave name and the FLUX name of your
	VOTABLE spectrum, e.g., “WAVE” and “FLUX”.
	These are the names that correspond to your wavelength and flux
	fileds in the votable respectively. See our IRAS_votable.xml

2. DATA IN VO FORMAT


 


For the time being, VOSpec only accepts data in FITs
binary tables and VOTable formats. If you have some data in ASCII
with proper information, it should be very easy to convert them into
VOTable. We will give an example below. If you have your data
available in FITS binary tables, then you don’t have to format
them any further (you will have to give the AXES metadata so that
VOSpec knows where to look at in the table to find the wavelength and
flux columns).


 


So for the time being, VOSPec only understand simple
“two column” spectra: one column for wavelength, and
another one for flux. Whenever other more complex formats are agreed
by the IVOA community, we will adapt the VOSpec to understand them.
